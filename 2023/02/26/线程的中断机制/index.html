Java线程中断机制与LockSupport

发表于 2023-02-26
  • Arvin.Yang
首页

​ 一个线程不应该由其他线程来强制中断或停止，为了线程的健壮性，而是应该由线程自己自行停止，自己决定自己的命运。所以，Thread.stop，Thread.suspend，Thread.resume 都已经废弃了。

​ 在 Java 中没有办法立即停止一条线程，然而停止线程却显得尤为重要，如取消一个耗时操作。因此，Java 提供了一种用于停止线程的协商机制 ---- 中断，也即中断标识协商机制，并非强制。

​ 中断只是一种协商机制，Java 没有给中断增加任何语法，中断的过程完全需要程序员自己实现。若要中断一个线程，你需要手动调用该线程的 interrupt 方法，该方法也仅仅是将线程对象的中断标识设置成 true。

public void interrupt() {
  	if (this != Thread.currentThread())
    	checkAccess();
  	synchronized (blockerLock) {
    	Interruptible b = blocker;
    	if (b != null) {
      		// 具体还是底层的系统方法，给该线程的状态打上标识。
      		interrupt0();           // Just to set the interrupt flag
      		b.interrupt(this);
      		return;
    	}
  	}
  	interrupt0();
}

​ 接着你需要自己写代码不断地检测当前线程的标识位，如果为 true，表示别的线程请求这条线程中断。此时究竟该做什么需要你自己写代码实现。

​ 每个线程对象中都有一个中断标识位，用于表示线程是否被中断；该标识位为 true 表示中断，为 false 表示未中断；通过调用线程对象的 interrupt 方法将该线程的标识位设置为 true；可以在别的线程中调用，也可以在自己的线程中调用。

1、Thread 线程类中的 interrupt () 方法

实例方法：仅仅设置线程的中断状态为 true，发起一个协商而不会立刻停止线程。

2、Thread 线程类中的 interrupted () 方法

静态方法：判断线程是否被中断，并且清除当前的中断状态。isInterrupted (true)

第一：返回当前线程的中断状态，（测试当前线程是否被中断）。

第二：将当前线程的中断状态清零并重新设置为 false，（清除线程的中断状态）。

3、Thread 线程类中的 isInterrupted () 方法

判断当前线程是否被中断（通过检查中断标识位，不会重置。）

4、如何中断一个运行中的线程？

基本原理：通过一个线程可见性变量值的变化，来控制根据该变量值来控制执行逻辑的线程。

​ 方式 1：通过一个 volatile 变量实现，例如：A 线程想停止 B 线程，那么 A 线程需要持有 B 线程的信号变量（volatile），B 线程监控着该变量的的变化情况，做出响应，比如结束自己。

​ 方式 2：通过原子类变量（类似方式 1）。

​ 方式 3、通过线程的中断标识协商机制。

5、如果停止一个运行中的线程？

通过上面一些方法，我们可以终止一些正常运行的线程，当某个线程中，调用了 wait，join 等方法会处于阻塞中，那么调用 interrupt 方法，那么线程会被唤醒，并且将线程的中断状态还原以及抛出异常。所以在线程中很可能出现死循环的等问题，无法结束线程操作，解决方案：实在捕获异常后，需要重新调用 interrupt 方法，因为 wait 方法接受中断信号号，会还原线程的中断状态。

7、3 种让线程等待和唤醒的方法。

方式 1、使用 Object 中的 wait () 方法让线程等待，使用 Object 中的 notify () 方法唤醒线程

方式 2、使用 JUC 包中的 Condition 的 await () 方法让线程等待，使用 signal () 方法唤醒线程

方式 3、LockSupport 类可以阻塞当前线程以及唤醒指定被阻塞的线程。此方法采用许可证思想。

许可证 permit 默认没有不能放行，所以一开始调用 park 方法当前线程就会被阻塞，直到别的线程给当前线程发放 permit，park 方法才会被唤醒。重点：许可证不能累计

这三种方式的进化过程：

第一种，缺点：1、wait，notify 方法必须要在同步块或者方法里面，且成对出现使用。

​ 2、先 wait，后 notify 才 OK，否则会出现线程永远阻塞。

第二种，解决了：第一种的那两个问题都没有解决

​ 缺点：1、await，signal 方法必须要在要先获取锁，且成对出现使用。

​ 2、先 await，后 signal 才 OK，否则会出现线程永远阻塞。

第三种，前两种缺点都解决了。

​ 无锁块，无顺序限制。

LockSupport 总结：

LockSupport 是用来创建锁和其他同步类的基本线程阻塞原语。

LockSupport 是一个线程阻塞工具类，所有的方法都是静态方法，可以让线程在任意位置阻塞，阻塞之后也有对应的唤醒方法。归根揭底，LockSupport 调用的 Unsafe 中的 native 代码。

LockSupport 提供了 park ()，和 unpark () 方法实现阻塞线程和解除线程阻塞的过程。

LockSuport 和每个使用它的线程都有一个许可证（permit）关联。

每个线程都有一个相关的 permit，permit 最多只有一个，重复调用 unpark 也不会积累凭证。
